# Deployment Comparison: Local vs Server

## Overview

Sandcastle has two deployment methods:
1. **Local Development** (`mise run deploy:local`) - Docker Compose with local build
2. **Server Production** (`installer.sh install`) - Parameterized deployment with Dockyard

## Detailed Comparison Table

| Aspect | `deploy:local` (docker-compose.local.yml) | Server Deploy (installer.sh) |
|--------|-------------------------------------------|------------------------------|
| **Compose File** | `docker-compose.local.yml` | Generated at `$SANDCASTLE_HOME/docker-compose.yml` from `installer/templates/docker-compose.yml.template` |
| **Image Source** | Build locally: `sandcastle:local` | Pull from registry: `${APP_IMAGE}` (default: `ghcr.io/thieso2/sandcastle:latest`) |
| **Build Args** | `BUILD_VERSION`, `BUILD_GIT_SHA`, `BUILD_GIT_DIRTY`, `BUILD_DATE` | N/A (uses pre-built images) |
| **Runtime** | Default Docker runtime | `runtime: runc` explicitly set (avoids Sysbox conflicts) |
| **Ports** | `8080:80`, `8443:443` | `${SANDCASTLE_HTTP_PORT}:80` (default 80), `${SANDCASTLE_HTTPS_PORT}:443` (default 443) |
| **Hostname** | Hardcoded: `sandcastle.local` | Variable: `${SANDCASTLE_HOST}` |
| **Data Storage** | Named volumes: `sandcastle-data`, `traefik-data`, `pgdata` | Host directories: `${DATA_MOUNT}` (typically `/sandcastle/data`) |
| **Postgres Data** | Named volume: `pgdata` | Host directory: `${SANDCASTLE_HOME}/data/postgres` |
| **Traefik Config** | Inline via `init-traefik` container | External files: `${DATA_MOUNT}/traefik/traefik.yml`, `${DATA_MOUNT}/traefik/dynamic/` (generated by installer) |
| **TLS Certificates** | Copied from `/tmp/sandcastle-traefik-certs/` | Self-signed or Let's Encrypt, stored in `${DATA_MOUNT}/traefik/certs/` |
| **Docker Socket** | `/var/run/docker.sock` | `${DOCKER_SOCK}` (typically points to Dockyard socket: `/sandcastle/docker.sock`) |
| **Docker GID** | `${DOCKER_GID:-0}` (default: root group) | `${DOCKER_GID:-988}` (sandcastle group on host) |
| **Network** | `sandcastle-web` (external, created manually) | `sandcastle-web` (external, created by installer from Dockyard pool) |
| **Init Containers** | `init-traefik` (Traefik setup), `init-data` (data dirs) | None (installer creates dirs/files before compose up) |
| **Worker Command** | `bundle exec rake solid_queue:start` | `./bin/jobs` |
| **Admin Setup** | Hardcoded in `migrate` service env vars | From `${SANDCASTLE_ADMIN_*}` variables |
| **OAuth Support** | Not configured | `${GITHUB_CLIENT_ID}`, `${GITHUB_CLIENT_SECRET}`, `${GOOGLE_CLIENT_ID}`, `${GOOGLE_CLIENT_SECRET}` |
| **Terminal URL** | `SANDCASTLE_TERMINAL_URL: https://sandcastle.local:8443` | Not set (defaults to SANDCASTLE_HOST) |
| **Database Password** | `${DB_PASSWORD:-sandcastle}` (default fallback) | `${DB_PASSWORD}` (generated by installer, stored in `/sandcastle/data/postgres/.secrets`) |
| **Variable Escaping** | Regular: `${VAR}` | Template syntax: `\${VAR}` (literal $ in generated file) |
| **Dockyard Integration** | ❌ Uses host Docker daemon | ✅ Uses isolated Dockyard Docker daemon with Sysbox |
| **Sandbox Isolation** | ⚠️ Sandboxes on same daemon as app | ✅ Sandboxes on separate Dockyard daemon |
| **Startup Command** | `mise run deploy:local` | `sudo installer.sh install` (or `reset`, `uninstall`) |
| **Prerequisites** | Docker, Compose, certs in `/tmp/` | Docker, Sysbox, root access |
| **User/Group** | Runs as default Docker user | Runs as `sandcastle:sandcastle` (UID/GID from installer config) |
| **Service Restarts** | Manual: `docker compose -f docker-compose.local.yml restart web` | Systemd (optional) or manual via Dockyard docker |
| **Logs Access** | `docker compose -f docker-compose.local.yml logs -f` | `$SANDCASTLE_HOME/docker-runtime/bin/docker-logs` wrapper |
| **Cleanup** | `mise run deploy:local:reset` | `sudo installer.sh uninstall` or `reset` |

## Key Architectural Differences

### 1. Docker Daemon Isolation

**Local:**
- Single Docker daemon
- App and sandboxes share same daemon
- No Sysbox, no container runtime isolation

**Server:**
- **Dockyard**: Separate Docker daemon (`sc_docker`) with Sysbox as default runtime
- App containers use standard `runc` runtime
- Sandbox containers use `sysbox-runc` runtime for Docker-in-Docker isolation
- App talks to Dockyard via socket: `/sandcastle/docker.sock`

### 2. File System Layout

**Local:**
```
/var/run/docker.sock         # Host Docker socket
/tmp/sandcastle-traefik-certs/  # Local certs
Docker volumes:
  - sandcastle-data
  - traefik-data
  - pgdata
```

**Server:**
```
/sandcastle/                  # SANDCASTLE_HOME
  ├── docker-compose.yml      # Generated from template
  ├── .env                    # Runtime secrets
  ├── etc/
  │   ├── sandcastle.env      # Installed config
  │   ├── dockyard.env        # Dockyard config
  │   └── postgres/
  │       └── init-databases.sh
  ├── data/
  │   ├── traefik/
  │   │   ├── traefik.yml     # Generated config
  │   │   ├── dynamic/        # Route configs
  │   │   ├── acme.json       # Let's Encrypt certs
  │   │   └── certs/          # Self-signed certs
  │   ├── postgres/           # Postgres data
  │   ├── users/              # Per-user home dirs
  │   ├── sandboxes/          # Sandbox volumes
  │   ├── wetty/              # WeTTY SSH keys
  │   └── ...
  ├── docker-runtime/
  │   └── bin/
  │       └── docker-logs     # Wrapper script
  └── docker.sock             # Dockyard socket
```

### 3. Traefik Configuration

**Local:**
- Traefik runs with inline CLI args
- Routes defined in init container script
- Single `sandcastle.local` host
- Self-signed certs only

**Server:**
- Traefik reads config from file: `traefik.yml`
- Routes in `dynamic/` directory (Rails generates)
- Supports Let's Encrypt or self-signed
- Multi-host support (public + Tailscale)
- Dynamic route updates (for sandbox routing)

### 4. Security & Isolation

**Local:**
- Suitable for development/testing
- No multi-tenancy isolation
- Sandboxes share Docker daemon with app
- SSH keys in init container

**Server:**
- Production-grade isolation via Dockyard + Sysbox
- Each sandbox in isolated Sysbox container
- Separate Docker daemon prevents escape
- SSH keys in persistent host directories
- Admin account required for first boot

### 5. Lifecycle Management

**Local:**
```bash
# Start
mise run deploy:local

# Stop
mise run deploy:local:down

# Reset (remove volumes)
mise run deploy:local:reset

# Logs
mise run deploy:local:logs
```

**Server:**
```bash
# Install
sudo installer.sh install

# Update (reinstall preserving data)
sudo installer.sh reset && sudo installer.sh install

# Uninstall (remove everything)
sudo installer.sh uninstall

# Logs
/sandcastle/docker-runtime/bin/docker-logs
```

## When to Use Each

### Use `deploy:local` when:
- Developing Sandcastle itself
- Testing changes quickly
- No need for Sysbox isolation
- Running on macOS/Windows (Dockyard requires Linux)
- Single-user testing

### Use `installer.sh` when:
- Production deployment
- Multi-user environment
- Need Docker-in-Docker isolation (Sysbox)
- Linux server
- Persistent data required
- Public internet access

## Migration Path

**From Local to Server:**
1. Export admin credentials and data
2. Install Dockyard and Sysbox on Linux server
3. Run `installer.sh gen-env` to create config
4. Edit `sandcastle.env` with your settings
5. Run `sudo installer.sh install`
6. Import data if needed

**No direct migration:** The two deployments are intentionally separate. Local is ephemeral (volumes), Server is persistent (host directories).
