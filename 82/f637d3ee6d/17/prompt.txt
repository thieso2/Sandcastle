### 1. **Podman Inside a Container: Kernel Features & Capabilities**

**Rootless Podman in Sysbox containers:**
- Rootless Podman can run inside a Sysbox container, leveraging Sysbox's user-namespace isolation. Sysbox provides "VM-like capabilities" allowing unprivileged users to run system-level applications like Docker and Kubernetes securely from workspace containers.
- **Key requirement:** Sysbox enables containers to run without the `--privileged` flag while still supporting system software.

**Rootless Podman in plain --privileged runc container:**
- Technically possible but defeats the security purpose of rootless mode. Privileged runc containers break isolation completely—the container gains full host device access, kernel module loading, and can exploit cgroup release_agent escape vectors to gain host root.

**Kernel features needed for rootless Podman (regardless of outer runtime):**
- User namespaces (mapped to unprivileged host UIDs via `/etc/subuid`)
- Network namespaces (for slirp4netns or pasta)
- Mount namespaces
- Capabilities: minimal set (no SYS_ADMIN required as of 2025 with `--isolation chroot` + Buildah improvements)
- FUSE support for fuse-overlayfs (or native overlayfs if kernel supports user-ns mapping)

**2025 improvement:** Kernel-level enhancements now allow rootless Podman to operate without SYS_ADMIN using container isolation techniques, matching Docker native overlay performance.

- Bind-mount source should be owned by range [165536:231071] (sysbox subuid range)
- Only Ubuntu ships shiftfs kernel module; other distros require manual ownership setup
- Normal bind-mount (no shiftfs) used if directory already owned by sysbox uid range (avoids overhead)

## Topic 1: Podman Rootless Inside Containers

### Kernel & Capability Requirements

Running Podman inside a container (nested) requires different configurations depending on whether the inner Podman runs rootful or rootless:

**Rootful Podman inside a container (without `--privileged`):**
- `--cap-add=sys_admin,mknod` — CAP_SYS_ADMIN is needed to mount filesystems, CAP_MKNOD to create devices under `/dev`
- `--device=/dev/fuse` — exposes FUSE to the container so fuse-overlayfs can function
- `--security-opt label=disable` — disables SELinux, which otherwise blocks required filesystem mounts

**Rootless Podman inside a container (without `--privileged`):**
- Does NOT require CAP_SYS_ADMIN or CAP_MKNOD
- Still requires `--device=/dev/fuse` (unless running kernel >= 5.12 with native overlay)
- Requires `--security-opt label=disable` and `--security-opt unmask=ALL`
- Requires `/etc/subuid` and `/etc/subgid` with enough UID/GID range entries for the user namespace, and `newuidmap`/`newgidmap` executables with proper file capabilities (`cap_setuid`, `cap_setgid`)
- The container running Podman must have CAP_SETUID and CAP_SETGID within its own user namespace

**Example command (rootful, no `--privileged`):**
```
podman run --cap-add=sys_admin,mknod \
           --device=/dev/fuse \
           --security-opt label=disable \
           quay.io/podman/stable \
           podman run ubi8-minimal echo hello
```

### Native Overlay vs fuse-overlayfs

- **Kernel < 5.12 or Podman < 3.1:** Must use fuse-overlayfs (hence the `/dev/fuse` device requirement)
- **Kernel >= 5.12, Podman >= 3.1:** Native unprivileged overlayfs is available (introduced in 5.11, SELinux bug fixed in 5.13). Once native overlay is used, `/dev/fuse` is no longer required at all
- Arch Linux confirmed: fuse-overlayfs is no longer a required dependency with Linux >= 5.11
- Networking in rootless mode uses `pasta` (default since Podman 5.0) or `slirp4netns`, both of which operate in user space — slower than root-mode networking

### The `--privileged` Path

`--privileged` works but is the least secure option: it grants all capabilities and removes seccomp/AppArmor/SELinux restrictions. The preferred approach for production is the explicit capability grant described above.

### Critical Open Issue: Docker 28.5.2 / runc 1.3.3 Breaks Sysbox

**GitHub Issue #973** (opened November 7, 2025, status: UNRESOLVED) documents a severe breakage:

- Docker 28.5.2 ships runc v1.3.3 which includes security patches for CVE-2025-52881 (procfs write redirect)
- runc v1.3.3 added protections that detect "unsafe" `/proc` overmounts and reject them with `EXDEV`
- Sysbox virtualizes `/proc` and `/sys` — runc now detects these as unsafe cross-device operations and aborts container init
- Error: `"error during container init: open sysctl net.ipv4.ip_unprivileged_port_start file: unsafe procfs detected: openat2 /proc/./sys/net/ipv4/ip_unprivileged_port_start: invalid cross-device link"`
- **Workaround:** Pin `containerd.io` to version 1.7.28-1 (not a production-ready fix)
- **No fix from Sysbox maintainers as of research date.** The runc team confirms this is intentional security behavior. Sysbox would need to stop virtualizing `/proc` or patch their own runc fork to remove the CVE protection — both architecturally problematic

This is a serious compatibility cliff: any host running Docker >= 28.5.2 (which ships runc 1.3.3) will have a broken Sysbox installation with no upstream fix available.

### Other Recent Issues

- **Issue #892:** Sporadic filesystem operation failures with Docker 26/27 inside Sysbox containers (~1 in 10,000 rate); workaround was rolling back containerd to 1.7.23, fixed in Sysbox 0.6.6
- **Issue #866:** `docker cp` into Sysbox containers results in `nobody:nogroup` owned files on Ubuntu 24.04 (the same ownership-shift problem Sandcastle hit)
- **Issue #946:** Inner Docker network connectivity failures on Ubuntu 24.04 + Kubernetes 1.32
- **Issue #849:** ZFS + Sysbox breakage

## Topic 3: Podman Socket Docker API Compatibility

### How `podman system service` Works

Podman exposes two socket endpoints:
1. A **Docker compatibility layer** implementing Docker API v1.40/1.41
2. A **libpod-native layer** with Podman-specific extensions

The socket is activated via `systemctl --user enable --now podman.socket` (rootless) or the system-level equivalent. Tools that set `DOCKER_HOST` to the Podman socket path can use it as a drop-in replacement — with important caveats.

### Major API Compatibility Gap (2024-2025): API Version Floor

**Docker client v29 requires a minimum server API version of 1.44.** Podman currently caps its compatibility layer at API version **1.41**. This is documented in:
- [GitHub Discussion #27877](https://github.com/containers/podman/discussions/27877): "API version 1.41 is not supported by this client: the minimum supported API version is 1.44"
- [Issue #27890](https://github.com/containers/podman/issues/27890): "Docker client v29 now requires Docker server v25"

This means: any tool that ships or uses Docker CLI v29+ will fail against the Podman socket. The gap is 3 API minor versions behind.

### BuildKit Incompatibility

`docker build` with BuildKit (`DOCKER_BUILDKIT=1`, which is the default in recent Docker) fails against Podman because:
- The `/v1.41/build/cancel` endpoint is missing from Podman's compat layer
- BuildKit also tries to upgrade the connection to h2c (HTTP/2 cleartext) and receives 404
- Error: `"failed to dial gRPC: unable to upgrade to h2c, received 404"`
- [Issue #17836](https://github.com/containers/podman/issues/17836): `docker build` fails with BuildKit due to missing API

### VS Code Remote Containers

- Works, but requires manual configuration: set `DOCKER_HOST` or configure the Docker extension to use the Podman socket path
- Rootless socket path (`$XDG_RUNTIME_DIR/podman/podman.sock`) differs from Docker's `/var/run/docker.sock` — some tools hardcode the Docker path
- [Issue #10005 (VS Code)](https://github.com/microsoft/vscode-remote-release/issues/10005): rootless socket path mismatch
- [Issue #17306 (containers/podman)](https://github.com/containers/podman/issues/17306): VS Code Docker extension v1.23.0 broke Podman compatibility
- Red Hat published a 2023 guide covering the workarounds: [Remote container dev with VS Code and Podman](https://developers.redhat.com/articles/2023/02/14/remote-container-development-vs-code-and-podman)

### kind / k3d

- **kind:** No official Podman support documented; kind relies on Docker socket and Docker-specific behaviors
- **k3d:** Podman support is **explicitly labeled experimental** as of 2024; requires either symlinking the Podman socket to `/var/run/docker.sock` or exporting `DOCKER_HOST` and `DOCKER_SOCK`; not guaranteed to work
- [k3d Podman docs](https://k3d.io/v5.8.3/usage/advanced/podman/)

### General Conclusion

Podman's Docker socket emulation works well for simple `docker run`, `docker ps`, `docker images` use cases and basic CI scripts. It breaks down for: newer Docker CLI clients (v29+), BuildKit-enabled builds, tools that rely on Docker's streaming APIs or undocumented behaviors, and anything requiring API > 1.41.

## Topic 4: fuse-overlayfs Performance vs overlay2

### The Hierarchy of Storage Drivers

From fastest to slowest in nested container contexts:
1. **Native overlay2** (kernel overlay on BTRFS/XFS host) — fastest, requires kernel-level privilege or kernel 5.12+ unprivileged support
2. **Native rootless overlay** (kernel 5.12+, unprivileged) — ~same speed as above for runtime, slightly slower container creation
3. **fuse-overlayfs** — 20-40% slower than native overlay2 for general I/O; user-space FUSE hop for every read/write
4. **vfs** — no CoW, complete layer copies on every new layer; extremely slow for image-heavy workflows but stable

### Quantified Performance Data

- **fuse-overlayfs vs native overlay2:** 20-40% slower generally; the FUSE userspace round-trip adds overhead for every I/O call
- **Extreme case (nested Podman-in-Podman builds):** One reported case ([Issue #401](https://github.com/containers/fuse-overlayfs/issues/401)) showed copying 130K files / 1.3 GB taking **over an hour** with fuse-overlayfs overlay driver vs **1.5 minutes** with vfs in that specific nested-container scenario. This is anomalous but illustrates pathological behavior under heavy small-file workloads in FUSE
- **Container creation time:** Native rootless overlay creates containers more slowly than fuse-overlayfs in some configurations, but does not affect runtime speed
- **Storage efficiency:** fuse-overlayfs is ~88% smaller storage footprint than vfs (CoW vs full copy)
- A 2024 Red Hat article confirms fuse-overlayfs outperforms vfs while being slower than native overlay for image builds in OpenShift Dev Spaces

### When vfs Is Forced

vfs fallback happens when:
- The host filesystem doesn't support overlayfs (e.g., NFS, some CIFS mounts)
- Running inside a container where the backing storage is itself an overlay layer (overlay-on-overlay) — the "upper fs missing required features" error
- No `/dev/fuse` access and kernel < 5.12 (no native rootless overlay either)

### Native Overlay in Rootless Mode (Modern Approach)

Since kernel 5.12 + Podman 3.1+, native unprivileged overlayfs eliminates the need for `/dev/fuse` entirely. Red Hat recommends preferring native overlay unless you have a specific reason (e.g., UID mapping requirements) to use fuse-overlayfs.

**Note:** Container creation with native overlayfs can actually be slower than fuse-overlayfs in some configurations (a known issue tracked in Fedora CoreOS issue #1749), though runtime performance is equivalent or better.

## Topic 5: Docker `--privileged` Security Alternatives

### What `--privileged` Actually Does

`--privileged` grants:
- All Linux capabilities (including CAP_SYS_ADMIN, CAP_NET_ADMIN, CAP_SYS_PTRACE, etc.)
- Disables seccomp filtering
- Disables AppArmor/SELinux confinement
- Makes all host devices accessible

A process that escapes the container filesystem has full root on the host. This is the attack surface that Sysbox eliminates.

### Sysbox Security Model: What It Avoids

Sysbox uses a Linux user namespace on every container with these specific protections:

| Property | --privileged + runc | Sysbox |
|----------|-------------------|--------|
| Root in container = root on host | Yes | No (maps to UID ~100000+) |
| All capabilities active on host | Yes | No (capabilities scoped to user namespace) |
| seccomp enforcement | Disabled | Active (Sysbox adds syscall interception) |
| AppArmor/SELinux | Disabled | Active |
| `/proc` write access to host | Yes | No (virtualized /proc) |
| Kernel module loading | Yes | Blocked (EPERM) |
| BPF modifications | Yes | Blocked |
| Cross-container privilege escalation | Possible | Prevented by unique UID ranges |

The key mechanism: even if a process inside a Sysbox container has all capabilities, those capabilities only apply to resources within the container's user namespace. Escaping the filesystem jail gives the attacker a UID in the range 100000-165535 with zero host privileges.

### Is `--privileged` + runc a Viable Secure Option? (2025 Verdict: No)

Three critical runc vulnerabilities disclosed in **November 2025** (affecting all runc versions prior to 1.2.8 / 1.3.3 / 1.4.0-rc.3) demonstrate the attack surface:

- **CVE-2025-31133:** `/dev/null` bind-mount replaced with symlink, allowing host file access during container init
- **CVE-2025-52565:** `/dev/console` bind-mount exploitation via pts symlink
- **CVE-2025-52881:** Race condition on shared mounts redirects runc writes to `/proc/sysrq-trigger` and other dangerous procfs files, enabling host crash or privilege escalation

These vulnerabilities affect `--privileged` containers particularly severely. They require ability to start containers with custom mount configurations, which any user with Docker socket access has.

**Note the painful irony for Sysbox:** the fix for CVE-2025-52881 (runc 1.3.3) added procfs safety checks that detect Sysbox's `/proc` virtualization as unsafe, breaking Sysbox on Docker 28.5.2 (Issue #973, documented above).

### Alternative Approaches Ranked by Security

1. **Sysbox (sysbox-runc):** Best isolation, no `--privileged` needed for DinD, uses user namespaces. Currently has a critical unresolved compatibility issue with Docker >= 28.5.2.
2. **Docker Enhanced Container Isolation (ECI):** Built on Sysbox, available in Docker Desktop for Business. Intercepts `--privileged` at the runtime level and runs it through Sysbox anyway.
3. **Rootless Docker daemon:** The entire dockerd runs unprivileged. Containers it spawns have reduced capabilities. But rootless Docker has networking limitations (no `--net=host` without workarounds, port binding quirks).
4. **gVisor (`runsc`):** User-space kernel intercepts all syscalls. Strong isolation but significant performance overhead and incomplete syscall coverage. Cannot run DinD.
5. **Kata Containers:** VM-based isolation. Strong security, full DinD support possible, but requires nested KVM or baremetal, higher overhead.
6. **`--privileged` + runc:** Do not use for multi-tenant or untrusted workloads. Acceptable only on single-tenant hosts where the container owner is trusted.

## Topic 6: nerdctl + containerd as Docker Replacement Inside Containers

### What nerdctl Is

nerdctl is a Docker-compatible CLI for containerd, developed by NTT. It is the closest 1:1 CLI replacement for the Docker CLI — commands are nearly identical (`nerdctl run`, `nerdctl build`, `nerdctl compose`). Unlike Podman (daemonless), nerdctl requires a running containerd daemon.

**GitHub:** [containerd/nerdctl](https://github.com/containerd/nerdctl)

### Rootless Support

nerdctl supports rootless mode via **RootlessKit** (a user-space network + UID mapping tool). The setup:
```
containerd-rootless-setuptool.sh install
systemctl --user start containerd
nerdctl run --rm hello-world
```

- Network in rootless mode uses slirp4netns or `bypass4netns` (nerdctl's accelerated variant, which can match rootful network speeds in some workloads)
- Rootless nerdctl uses `~/.local/share/containerd` for storage

### Running nerdctl Inside a Container (DinD)

This is harder than it looks. Unlike Sysbox which handles this transparently, running containerd inside a container faces:

1. **cgroup issues:** Rootless buildkitd inside a rootless nerdctl container fails with `runc run failed: no cgroup mount found in mountinfo` (Issue #3668, 2024). cgroup v2 delegation must be configured for unprivileged access.
2. **containerd daemon bootstrapping:** You need to run `containerd` as a process inside the container, which requires either root or user-namespace tricks.
3. **No official DinD image for nerdctl/containerd:** Unlike Docker's `docker:dind` image, there is no widely-maintained `nerdctl-dind` equivalent.

For true DinD with nerdctl, the practical path is running a rootful containerd inside a privileged or Sysbox container — which doesn't improve the security situation compared to Docker DinD.

### Compose Support and Gaps

nerdctl compose parses standard `docker-compose.yaml` files. Known gaps vs Docker Compose:

**Unimplemented flags:**
- `--resolve-image-digests`, `--no-interpolate`, `--format`, `--output`, `--profiles` (config subcommand)
- `--no-deps`, `--force-recreate`, `--always-recreate-deps`, `--no-recreate`, `--no-start`, `--abort-on-container-exit` (up subcommand)

**YAML field gaps:**
- `services.<SERVICE>.links` (unimplemented docker run flag)
- `services.<SERVICE>.build.extra_hosts`
- Various other fields that map to unimplemented nerdctl run/build flags

**Known bugs (2024):**
- `nerdctl compose` hangs in version 2.0 under certain conditions (Issue #3678)
- `nerdctl compose up` stuck on container creation (Issue #3520)
- Global compose lock prevents concurrent compose operations

### nerdctl vs Podman: Head-to-Head for DinD Use Cases

| Factor | nerdctl + containerd | Podman |
|--------|---------------------|--------|
| DinD support | Requires rootful containerd or privileged | Works with cap-add (no --privileged) |
| Rootless | Yes (RootlessKit) | Yes (native, daemonless) |
| Docker API socket emulation | No (no socket; uses containerd socket) | Yes (podman system service, but capped at API 1.41) |
| Docker Compose compat | Good, with gaps listed above | Good, via `podman compose` or `podman-compose` |
| Kubernetes CRI | Yes (containerd is a CRI runtime) | No |
| Image pull performance | Supports lazy pull / eStargz | Standard pull |
| WASM support | Yes (runwasi) | No |
| Daemon requirement | Yes (containerd must run) | No (daemonless) |
| Maturity for nested containers | Limited / needs work | Better documented |

### Practical Assessment

For a Sandcastle-like use case (DinD inside sandboxes), nerdctl+containerd does not offer a meaningful advantage over Podman. The main appeal of nerdctl is for Kubernetes-adjacent workloads where containerd is already the CRI, or for teams that want lazy image pulling (eStargz). For nested container scenarios, Podman's daemonless architecture is actually better suited since there is no daemon to keep alive inside each sandbox.
